# 项目功能设计

## 项目定位

MindAgent 是一个面向心理健康问答与智能体交互场景的后端项目，提供认证、会话、RAG 检索、管理员能力和本地联调页面。

## 核心功能模块

### 1. 认证与权限

- 用户登录
- Token 刷新
- 基于角色的接口访问控制
- 默认角色：
  - `ADMIN`
  - `USER`
  - `TEACHER`
- 独立登录页面用于统一进入管理员页或业务页
- 未登录访问用户页、管理员页、联调页时统一跳转到登录页
- 登录成功后按角色自动跳转：
  - `ADMIN` -> 管理员页
  - `TEACHER` -> 老师页
  - `USER` -> 用户页

### 2. 对话能力

- 流式聊天接口
- 意图识别
- QueryType 知识路由识别：
  - `PSYCHOLOGY_KNOWLEDGE`
  - `APPOINTMENT_PROCESS`
  - `APPOINTMENT_ACTION`
  - `OTHER`
- 多会话管理
- 会话消息存储
- 对话历史读取
- MySQL 长期摘要 + Redis recent window 短期记忆
- assistant 回复后异步压缩旧消息，保留最近 `6` 轮上下文
- 前端支持查看当前会话 Token 调试信息，包括摘要 token、recent window token、消息压缩统计与预计 prompt token
- AI 配置已拆成 `chat / agent / embedding / rerank`
- 当前默认聊天、预约 Tool Calling、Embedding、Rerank 全部走阿里云百炼
- 预约动作链路优先走 Spring AI 原生 Tool Calling；如果 Spring AI `ChatModel` 未注册，则降级到 DashScope/OpenAI-compatible `tools` 调用，仍复用同一组本地预约工具
- 已为后续“普通聊天切 Ollama、预约 Agent 保持百炼”预留独立配置
- 向量检索默认使用 `text-embedding-v4`

### 3. RAG 检索

- 本地知识库加载
- 应用层 BM25 检索
- Chroma 向量检索
- BM25 与向量结果在应用层做 RRF 融合
- RRF 融合后的候选集默认截取 Top30，再调用 Qwen `qwen3-rerank` 做二次排序，返回 Top5
- 检索后处理会执行去重、同文档 chunk 数限制、token 预算裁剪与低分过滤
- FAQ/KB 采用标题优先的递归切块策略，默认 chunk size 为 `900`、overlap 为 `120`
- 每个向量块都会写入结构化 metadata，用于区分 `faq` / `kb` 并保留章节、标签、风险等级、受众等信息，同时统一补充 `knowledge_base_key`、`document_id`、`chunk_index`
- Chroma 重载采用整库重建，先清空旧向量块再重新写入当前知识文件
- 管理员触发知识库重载
- 管理员上传文档采用异步入库流程：
  - 保存原始文件
  - 写入 `kb_document`
  - 后台解析文档并切分 chunk
  - 写入 `kb_chunk`
  - 文档状态先在数据库事务中更新为 `ACTIVE`
  - 向量化后写入 `vector_store`，不阻塞文档状态提交
  - 这样即使向量同步较慢，也不会让页面长期停留在 `PROCESSING`

### 4. 管理能力

- 管理员查看报告
- 管理员查看调度日志
- 管理员执行知识库相关操作
- 管理员可通过独立入口分别上传 FAQ 文档与 KB 文档
- 管理员可通过两个知识库空间分别上传文档：
  - `上传校园心理健康知识`
  - `上传校园内部知识库`
- “上传校园内部知识库” 上传的文档统一保存到 `knowledge_base_3`
- 校园心理健康知识空间已合并原来的权威科普资料内容
- 每个知识库空间下再按 `faq/` 与 `kb/` 两个子目录组织文件，上传时会按空间与类型落盘
- 管理员可通过文档选择列表切换当前查看的文档
- 同名文档上传会覆盖旧 chunk，并重新生成新的 chunk
- 管理员可通过文档状态接口查看 `PROCESSING / ACTIVE / FAILED`
- `GET /api/kb/documents/{id}` 用于查询知识文档处理状态与元数据
- `GET /api/kb/documents` 用于返回文档列表
- `DELETE /api/kb/documents/{id}` 用于软删除文档
- 管理员知识库页面展示：
  - 文档列表
  - 原始文档信息
  - 切分后的 chunk 列表
  - chunk metadata
  - 是否已写入向量库
- 管理员前端按三栏工作台组织：
  - 左侧管理操作与上传入口
  - 中间文档列表、状态筛选与搜索
  - 右侧文档详情、原始信息与 chunk 明细

### 5. 预约闭环

- 老师资料管理：
  - `GET /api/teacher/profile/me`
  - `PUT /api/teacher/profile/me`
- 老师排班管理：
  - 老师先录入某天几点到几点有空，系统自动按 1 小时切分生成排班表明细
  - `GET /api/teacher/slots`
  - `POST /api/teacher/slots`
    - 入参仍为 `startTime / endTime / location / notes / open`
    - `startTime / endTime` 必须整点
    - 时长必须为 1 小时整数倍
    - 若与现有任意排班重叠，则整次创建失败
  - `PUT /api/teacher/slots/{id}`
    - 仅编辑单个 1 小时排班
  - `DELETE /api/teacher/slots/{id}`
  - `PATCH /api/teacher/slots/{id}/open`
  - `PATCH /api/teacher/slots/{id}/close`
  - `BOOKED` 排班不可编辑、不可删除、不可开关
- 学生预约能力：
  - `GET /api/appointment/slots/available`
    - 仅返回未来的 `AVAILABLE` 时段
    - 前端按老师分组展示老师与对应空闲时段
  - `POST /api/appointment/create`
  - `GET /api/appointment/my`
  - `POST /api/appointment/{id}/cancel`
- 老师预约管理：
  - `GET /api/teacher/appointments`
  - `GET /api/teacher/appointments/{id}`
  - `PATCH /api/teacher/appointments/{id}/cancel`
  - `PATCH /api/teacher/appointments/{id}/complete`
- 老师端前端按双栏工作台组织：
  - 左侧为个人资料与空闲时间 / 排班表维护
  - 右侧为预约列表、详情、取消与完成操作
- 创建预约采用“创建即确认”：
  - slot：`AVAILABLE -> BOOKED`
  - appointment：`CONFIRMED`
- 取消预约时：
  - appointment：`CANCELLED`
  - slot：`BOOKED -> AVAILABLE`
- Tool Calling 提供：
  - `queryAvailableSlots`
    - 支持 `date / teacherName` 参数
    - 返回 `teacherUserId / teacherDisplayName / nextAvailableTime / slots`
  - `createAppointment`
  - `queryMyAppointments`
  - `cancelAppointment`
- 所有工具调用记录到 `ai_tool_call_log`

### 6. MCP 模拟能力

- Excel 写入模拟接口
- Email 发送模拟接口
- 调度结果记录

## 数据存储设计

### MySQL

主要业务表：

- `app_users`
- `chat_session`
- `chat_message`
- `chat_session_memory`
- `chat_memory_compress_log`
- `mcp_dispatch_logs`
- `psychological_reports`
- `refresh_tokens`
- `kb_document`
- `kb_chunk`
- `teacher_profile`
- `teacher_available_slot`
- `counseling_appointment`
- `appointment_log`
- `ai_tool_call_log`

初始化来源：

- 首次 Docker 初始化执行 `backend/src/main/resources/db/init_schema.sql`
- `psychological_reports.query_text` 使用 `LONGTEXT`，与实体中的 `@Lob + columnDefinition = "LONGTEXT"` 保持一致

后续增量：

- 手工维护 `backend/src/main/resources/db/migration/`

### Redis

- 用于最近窗口消息短期缓存
- key：`mindagent:chat:recent:{userId}:{sessionId}`
- value：最近窗口消息列表，默认 TTL `24h`

### Chroma

- 用于知识库向量检索
- 当前知识库入库时会将 FAQ/KB 的结构化章节写入向量库，并通过 metadata 保留 `doc_name`、`section_title`、`category`、`tags`、`risk_level`、`audience`、`version` 等字段
- 启动时会检查预约流程文档是否误放在 `knowlegde_base_1`，若命中则迁移到 `knowledge_base_3` 并重建 chunk 与向量索引
- 校园心理健康知识空间已合并原来的权威科普资料内容，统一从 `knowledge_base_1` 入口管理
- 向量化模型默认配置为 `text-embedding-v4`，可通过 `SPRING_AI_OPENAI_EMBEDDING_MODEL` 覆盖

### 本地文件存储

- `backend/storage/knowledge`：知识库原始文本目录
- `backend/storage/knowledge/knowlegde_base_1/faq|kb`：校园心理健康知识空间
- `backend/storage/knowledge/knowledge_base_3/faq|kb`：校园内部知识库空间
- `backend/storage/knowledge/knowledge_base_2` 已删除，原权威科普资料并入 `knowledge_base_1`
- Chroma 仅保存向量化结果，不替代原始知识文件目录

## 本地运行设计

- 根目录 `docker-compose.yml` 统一拉起 MySQL、Redis、Chroma、nginx
- 后端服务默认运行在 `8082`
- 前端 nginx 默认运行在 `8088`

## 聊天接口设计

- `POST /api/chat/stream`
  - 请求体：`sessionId + query + model`
  - 后端上下文：`system prompt + session summary + recent messages + current query + optional RAG`
  - 先做 `IntentType` 分类，再做 `QueryType` 路由；`OTHER` 不走 RAG
  - `APPOINTMENT_PROCESS` 继续走 RAG 解释预约规则
  - `APPOINTMENT_ACTION` 走预约 Tool Calling 链路，优先 Spring AI，必要时降级到 OpenAI-compatible `tools`
  - SSE 事件保留 `intent / rag / token / done`，并新增 `tool_call_start / tool_call_result / final_reply`
- `POST /api/rag/retrieve/debug`
  - 返回 `intentType`、`queryType`、`rewrittenQuery`
  - 返回 `bm25Results / vectorResults / fusedResults / rerankResults / selectedResults`

## RAG 调试展示

- 用户前端 `frontend/index.html` 提供“调试召回”按钮
- 调试面板展示：
  - `IntentType / QueryType`
  - `BM25 / Vector / Fused / Rerank / Selected` 数量
  - 最终 Top5 chunk 的文档名、chunkId、知识库空间、分类、来源类型与内容预览
- `POST /api/chat/sessions`
  - 创建新会话
- `GET /api/chat/sessions`
  - 返回当前用户会话列表
- `GET /api/chat/sessions/{sessionId}/messages`
  - 返回该会话完整消息历史
- `GET /api/chat/sessions/{sessionId}/token-debug`
  - 返回当前会话 token 调试信息
  - 支持传入 `draftQuery` 估算当前输入参与 prompt 后的 token 结果
- `DELETE /api/chat/sessions/{sessionId}`
  - 删除会话、消息、摘要与 recent cache

## 记忆分层设计

- MySQL 是唯一持久化真相层
- Redis 只缓存最近窗口，不缓存整会话、不缓存长期摘要
- recent cache 读取顺序：
  - 先读 Redis
  - 未命中则回源 MySQL
  - 将裁剪后的最近窗口回填 Redis
- 压缩触发条件：
  - 未压缩历史 token 超过 `3000`
  - 或未压缩消息数达到约 `12` 轮阈值

## 文档维护要求

- 功能有新增或调整时，必须同步更新本文件
- 数据库结构变更时，必须同步更新 `init_schema.sql`
