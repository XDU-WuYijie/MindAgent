普通RAG不是必须走知识库吗，为什么这里可以不查知识库，因为我们这里有意图判断，相当于一个路由RAG，那意图判断怎么实现的呢？

我们给大模型一个专门的分类 Prompt ，强制它对用户输入意图理解后只输出3种结果

CHAT：日常闲聊天气、吃饭、问候、娱乐
CONSULT：心理咨询、情绪问题、心理困扰
RISK：高风险表述自杀、自残、绝望、伤人

然后后端根据结果走分支，CHAT 则直接回，不记录 Excel、不预警，Consult、Risk则进入RAG 检索，记录 Excel

那Prompt怎么写的呢

你是一个用户意图分类器，只做意图识别，不回答问题。
用户输入内容：{user_input}

请将用户意图严格分为以下三类之一，只输出标签，不要其他任何内容：
- CHAT：日常闲聊、问候、天气、娱乐、无关内容
- CONSULT：心理咨询、情绪倾诉、压力、焦虑、低落、失眠、亲密关系、学习压力等心理相关
- RISK：自杀、自残、绝望、自伤、伤人、严重抑郁等高危内容

大模型只会返回CHAT / CONSULT / RISK 三个词之一给java后端

举个栗子子

今天天气怎么样？ → CHAT我最近好焦虑睡不着→ CONSULT我不想活了 → RISK你吃饭了吗 → CHAT我感觉很压抑怎么办 → CONSULT

学过算法的朋友应该就会觉得为什么这样比训练分类器更好呢？

首先这样不用额外训模型，直接用现有大模型就能做，然后Prompt 可随时调整，很灵活，且复杂度会低很多啊

项目是校园心理场景，意图类别固定，所以我采用大模型零样本分类实现意图判断，更简洁也更高效不是吗

SSE流式会话

Server-Sent Events（服务器推送事件），是一种基于 HTTP 的实时通信协议。大模型生成文本通常是逐字逐句输出的，而不是一次性生成完毕。

SSE 允许服务器将生成的内容像打字机一样源源不断地推送给前端浏览器或者App。这是实现类ChatGPT打字机效果的核心技术，相比 WebSocket 更轻量，适合单向流式传输。

MCP

  • Model Context Protocol模型上下文协议，由 Anthropic，也就是Claude 团队推出。解决大模型与外部数据源、工具之间的连接标准问题。通过 MCP，一个 LLM 可以统一接入各种 MCP 服务器， GitHub、本地文件系统这些，实现更安全的上下文交互，避免开发者需要为每一个数据源写定制化的代码。

SpringAI

Spring 社区推出的 AI 应用开发框架。它借鉴了 LangChain 的思维，但基于 Java/Spring 生态。它提供了统一的 API 来对接 OpenAI、Ollama、HuggingFace 等，封装了 Prompt 模板、RAG 流程、函数调用和 SSE 流式输出。

Ollama和OpenAI

本地部署 与 云端 API 两种使用大模型的方式。

  • Ollama是一个极简的本地大模型运行工具。让你可以一条命令在本地运行 Llama 3、Qwen、Gemma 等开源模型。

  • OpenAI提供闭源商用模型（GPT-4o, GPT-4o-mini 等）的云端 API。优势是性能强大、开箱即用，但需要网络访问且按 Token 计费。

Lora微调大模型

  • Low-Rank Adaptation低秩适应，一种轻量级大模型微调技术，不训练全量参数。

  • Lora 通过在原始模型旁路添加少量的可训练参数（适配器），冻结原模型权重。只需训练原模型参数量 0.1%-1% 的参数，就能让模型学会特定风格、特定知识或特定格式的输出。

  • 在 Transformer 层插入小的低秩矩阵，只训练这部分。显存占用小、速度快、保存的模型文件极小（几 MB～百 MB）。

  • 作用是让通用模型变成“心理顾问”或学习特定语气风格

RAG

  • Retrieval-Augmented Generation。在让 LLM 回答问题前，先从外部知识库（向量库）中检索相关的文档片段，将这些片段作为参考资料连同问题一起发给 LLM。

  • 核心作用是解决 LLM 的“幻觉”问题和知识滞后性。让模型可以回答私有数据（公司内部文档等）相关的问题。

动态路由RAG

动态路由 RAG 的本质就是先做意图判断，再决定是否走 RAG 检索，而不是无脑全量检索。传统RAG是单向的流水线，无论问题简单复杂都要去查知识库

Chroma向量库

一个开源的嵌入式向量数据库。在 RAG 流程中，需要将文本转换成向量并存储起来以便检索，向量库还有 Milvus、Pinecone 或 PostgreSQL 的 pgvector 插件。

Flux

  1. Facebook 提出的一种前端单向数据流架构，利用前端单向数据流架构来优雅地处理后端通过 SSE 持续推送的大模型生成内容。是实现了流式数据的接收与 UI 渲染解耦。

  2. 复杂的多模型、多会话场景下状态清晰可控，用户体验上也实现流畅的打字机效果。

Spring Security

Java 生态Spring 框架中的 安全框架。在构建 AI 应用后端时，不能直接将大模型接口暴露给所有人。Spring Security 负责处理 认证你是谁和 授权你能调用哪个模型？能查哪些向量库？防止未授权访问、API 滥用和提示词注入攻击。

九、🥛微调具体细节（重点）

前面我们已经讲了大致的原理，利用开源数据集离线微调Qwen2.5-7B模型并集成到Ollama里，供SpringAI调用

现在我们来讲一些细节

项目里 LoRA 微调 Qwen2.5-7B 具体怎么做？

  • 整个流程是离线微调，首先准备数据集，PsychQA 校园心理对话数据集，完全开源可商用，进行数据清洗、脱敏、格式化，让数据集变成微调格式。

  • 数据集弄好后，用 LoRA 脚本，冻结模型主干，只训练 Q/V 注意力层，这个在前置知识也有说。设置参数rank=8、alpha=16、学习率 2e-4，训练完以后呢，你会得到一个很小的LoRA 适配器文件，只有几十 MB。

  • 将基座模型也就是Qwen2.5-7B那个和我们得到的LoRA 适配器文件进行合并，并且导出为 GGUF 格式。然后把这个文件放到 Ollama 运行，供 SpringAI 调用

微调数据集是开源的吗？具体是哪个？

  • 用的PsychQA 校园心理对话数据集，纯开源，可商用

  • 这个数据集里有啥呢？心理咨询问答、情绪识别对话、焦虑、抑郁、压力疏导语料，这些都有，而且对于大家其实也知道心理对话其实有一定的版权和敏感性，这个数据集里的数据已经为我们做了脱敏、无隐私处理，那这些工作就不需要我们再去做了

  • 它大概有包含约 2000～3000 条高质量心理问答与情绪标注数据

离线微调要不要GPU?

  • 那肯定要啊兄弟们，没有GPU谈什么大模型，虽然Qwen2.5-7B是个小大模型（doge）

  • 因为我们数据量不多，2000~3000条，总体时间15分钟左右即可完成，我们用的单卡NVIDIA 3090 ti 24GB显存。

  • 然后我们冻结了大部分权重，只微调QV对吧，所以很快，如果你没有GPU，那你可以租云GPU，我是用的成电计算机学院组里实验室的，因为研究方向就在这一块

微调后的模型怎么集成、部署到 Ollama？

  • 前面讲了Qwen2.5-7B那个和我们得到的LoRA 适配器文件进行合并，并且导出为 GGUF 格式对吧，接着我们就写一个简单的modelfile，构建 Ollama 镜像，执行ollama create命令，生成自定义模型，部署到本地运行

  • 后面你就可以直接ollama run启动微调后的Qwen2.5-7B

十、🍫RAG细节（重点）

自动知识库构建怎么做的？

  • 管理员通过后台上传知识文档，包括心理咨询规范、常见问题解答，同时支持拉取指定代码库中Markdown或文本文件作为补充知识源。

  • 使用 SpringAI 集成的文本分割器TokenTextSplitter将文档切分为语义完整的块（chunks），避免截断关键信息。同时通过语义切分规则来保证 chunk 完整性，控制 chunk 大小为512 ，并且优先在段落、句子结尾切分，同时在检索时返回相邻 2-3 个 chunk 拼接上下文，间接实现类似overlap的效果.

  • 然后将得到的文本块通过Ollama或者OpenAI的text-embedding-3-small 模型转成向量，将向量与元数据一起存入Chroma向量数据库

检索增强生成流程

当用户发起对话时，首先将用户当前问题通过相同的text-embedding-3-small模型转为向量。然后在 Chroma 中执行 ANN（近似最近邻）搜索，返回最相关的top k个文本块。其中本项目中k=4

将检索到的文本块与对话历史合并，构造增强prompt。将增强提示发送给 微调后端qwen2.5-7B，基于知识库内容与对话状态生成更精准、合规的回答。

会话上下文如何进行管理呢？

利用 Flux 流式会话接口，维护每个会话的最近 10 轮对话记录，避免超出模型上下文长度。

具体实现

SpringAI 的 RAG 组件，利用 QuestionAnswerer 和 VectorStore 抽象，将 Chroma 作为 VectorStore 实现类。

Flux 流式响应则是在 Controller 层返回 Flux，实现逐 token 推送，同时可在流式过程中并行执行检索。

RAG知识库所准备的知识文档需要清洗吗？

要的，除了Lora微调的数据集是Open access的，其他部分数据集都是我们自己这边私有的。我们当时有对数据也就是实验室这边的心理对话对做重复部分、格式乱码、空行等做文本清洗，然后按主题分块，在送入text-embedding-3-small。

MCP 是协议规定模型怎么调用外部工具，Java 服务实现你写的发邮件、写 Excel

MCP在这个项目里怎么用的

告诉大模型有哪些工具可用，也就是写 Excel、发邮件，让模型输出结构化调用指令（JSON），然后后端执行工具，并把结果返回给模型

MCP如何调用写excel？

给大家解释一下区别

  • 普通rag是不管用户问什么，一律检索向量库 一律拼接知识生成回答，就是每次你问问题，管你问的啥，都必查向量库

  • 而我们是先让大模型做了意图判断，判断是闲聊还是心理问题or心理咨询，闲聊则直接回答，不走 RAG判断，是专业咨询才走 RAG 检索，这就是动态路由RAG。当然这和我另一个项目的agentic rag还是有差距，agentic rag可以多次检索多步推理，比动态路由rag强很多

说白了，动态路由rag就是在rag前面加了一个agent进行意图判断，这个怎么实现的我有讲过，通过prompt调教，强制让微调后的qwen2.5-7B给我输出Chat、Consult、Risk，大家可以结合上面的项目流程图看一下

这个项目你遇到的难点是什么？怎么解决的

  1. 两个异步 MCP 服务的数据一致性问题，qwen2.5模型通过流式对话分析出高风险状态后，需要同时通过MCP写Excel并发送邮件。但MCP服务是异步的外部调用，如果邮件发送成功但Excel写入失败，或者反之，就会导致数据不一致，比如说老师收到了邮件，但本地留档缺失。

  2. 解决方法通过状态机和重试机制保证最终一致性。给每一条高风险记录增加状态字段，记录 Excel 和邮件的执行状态。然后先写 Excel，成功后更新状态，再发送邮件。任何一步失败，让系统可以自动重试 3 次，避免网络波动导致的异常。如果最终失败的话，就记录异常并触发告警，支持后台手动补偿。

大模型调应用这一块MCP和Skills的区别你了解吗

你觉得MCP这个机制有什么问题？

  1. MCP 的通信是标准化的，但出错时错误信息不够友好，定位问题需要抓包分析。

  2. MCP 没有内置服务发现机制，需要手动配置 Server 地址，在 K8s 环境中需要配合 Service 注册。

  3. MCP 本身是无状态的，对于需要会话状态的操作，比如说 Excel 文件的持续写入，需要自己维护上下文。

  1. 两个 MCP 服务调用都会做异常捕获，失败后会记录错误日志，并更新 MySQL 里的状态字段，将风险表risk状态字段更新为发送失败，系统就能知道是邮件发失败了，还是 Excel 写失败了。

讲一下向量数据库的原理

向量数据库专门存储和检索高维向量。核心流程：

  1. 向量化，用Embedding模型将原始数据转为向量，长文档转为向量前需要分块，然后用嵌入模型对分好的块转成向量

  2. 索引构建，通过算法（HNSW、IVF）建立高效索引，加速近似最近邻搜索

  3. 相似度检索，给定查询向量，快速返回最相似的Top-K向量及对应元数据。常见指标包括召回率、查询延迟、可扩展性。

微调时选择了哪些层

主要针对Transformer的自注意力层。这些层负责语义交互和特征变换，微调它们能有效调整模型行为，而偏置项和LayerNorm保持冻结，参数量增加极少。

余弦相似度和欧氏距离在进行相似度匹配时，各自的优缺点是什么

  1. 余弦相似度衡量方向差异，忽略模长，适合文本等高维稀疏向量。

  2. 欧氏距离衡量绝对距离，若向量未归一化可能误判。若向量已L2归一化，两者等价。项目中使用余弦相似度，更符合语义比较。

向量化之前为什么要对长文档进行切片呢？如果不切片会有什么后果？

  1. Embedding模型有输入长度限制，超长文本会被截断丢失信息

  2. 长文档可能含多主题，整体向量化导致语义稀释

  3. 检索粒度粗糙，返回整篇文档需二次处理，增加开销和错误。若不切片，检索系统难以命中具体细节，影响下游生成质量。

agent中的长短期记忆了解吗？讲一下

Agent 中的长短期记忆核心是解决大模型上下文有限、无法跨会话持久化的问题

短期记忆：

短期记忆是当前会话的临时缓存，用完即丢，读写极快，仅当前会话或者任务有效，结束自动清空

短期记忆的实现，主要有滑动窗口、摘要压缩、状态追踪这些。滑动窗口呢比如说固定长度最近10轮，新的进旧的出。摘要压缩则是会话超长时用 LLM 把旧历史压缩成摘要，保留核心，省 token。

那短期记忆有什么作用呢？可以维持当前对话连贯、跟踪任务执行状态

上面已经讲完了短期记忆，接下来我们来讲一下长期记忆

长期记忆：

长期记忆是持久化存储、跨会话复用的结构化记忆，相当于 Agent 的经验库，重启、跨会话依然存在。但是不能直接读，要语义检索取出相关

长期记忆的技术实现，主要有向量数据库+ RAG 检索、混合存储Hybrid、知识图谱。向量数据库+ RAG 检索是把历史转为向量嵌入，存入向量库比如Chroma中，然后对输入向量余弦相似度检索 Top-K 相关记忆，拼进 Prompt 给 LLM。

那混合存储Hybrid呢？它是将结构化数据用户 ID、偏好、状态存到MySQL或者PostgreSQL中，语义、文本记忆放到向量库，原始日志放到文件、日志系统

那长期记忆有什么用呢？

可以进行经验复用，上次失败or成功→下次优化。同时也可以做长期规划，存目标、步骤、复盘，跨天执行复杂任务

你觉得RAG的瓶颈主要在哪

RAG目前的主要瓶颈包括：

  • 检索质量上限：即使优化检索，也可能召回不相关或遗漏关键信息，尤其是复杂推理问题

  • 上下文窗口限制：虽然LLM上下文不断增大，但填入大量检索块会增加成本和处理时间且稀释关键信息。

  • 分块与信息损失：固定分块可能破坏文档的全局结构，导致跨块信息无法关联。

  • 实时性与一致性：高频更新的知识库需要快速索引和同步

  • 成本与延迟：每个查询都需要检索和生成，相比纯LLM调用开销更大，难以在超大规模并发下保持低延迟。

即使说你用了RAG，但是你的agent可能给客户返回的一个结果，还有可能是不符合预期，或者简单说就是还是编造，有幻觉，那么出现这样的问题可能是什么原因？

  1. 召回率低：检索器未能从知识库中召回真正相关的文档片段，导致模型缺乏足够的事实依据，只能依赖自身参数知识来生成回答。

  2. 相关性排序偏差：即使召回了相关文档，但排序算法将不相关低质量的片段排在前面

  3. 知识库覆盖不全：用户问题涉及的知识不在知识库中，模型无法找到正确答案，可能强行编造。

  4. 文档存在噪声或错误：知识库中的文档本身包含错误、矛盾的表述，模型直接采纳这些信息生成答案。

  5. 上下文窗口限制：如果检索到的文档过长，模型可能被截断或忽略部分关键内容，导致信息丢失。

  6. 忽视检索内容：模型过度依赖其内部参数知识，而忽视或错误理解检索到的上下文

  7. 模型固有缺陷：即使有检索内容，生成式模型在组织语言时仍可能添油加醋，添加不在检索结果中的细节

  8. 过度自信：模型倾向于生成流畅的文本，即使面对不确定的信息，也会用看似合理的表述填充，造成幻觉。

什么是RAG检索增强生成？它的核心工作流程是怎样的？

RAG是一种将信息检索与大型语言模型生成能力相结合的技术。核心思想是在模型生成回答之前，先从外部知识库中检索出与用户问题相关的信息片段，将这些信息作为上下文与原始问题一起提供给LLM，有效缓解幻觉和知识滞后问题。 分为两个阶段：

  1. 索引阶段：首先对文档数据进行清洗、分割成合适的文本块，然后通过嵌入模型将每个块转换为向量，并将这些向量存储到向量数据库中，同时保留元数据。

  2. 查询阶段：用户提出问题后，同样用嵌入模型将问题向量化，在向量数据库中进行相似性搜索，召回最相关的k个文本块，将这些文本块与原始问题组合成增强提示，最后调用LLM生成最终回答。

RAG与大模型微调在解决大模型知识不足的问题上有什么本质区别？分别在什么场景下选用？

  • RAG是一种动态知识注入方式，模型不改变自身参数，而是通过检索实时获取外部知识，答案可追溯来源，更新知识只需更新数据库，成本低。

  • 微调是一种静态知识注入方式，通过调整模型参数让模型记住新知识，但知识固化在权重中，更新需要重新训练

场景选择：

  • RAG适合知识频繁更新需要引用来源、领域知识广泛且难以全部参数化。

  • 微调适合需要调整模型的行事风格、输出格式、特定术语使用（让模型模仿某种语气），或者让模型掌握不易通过检索获得的私有逻辑。

什么是"检索噪声"？怎么优化检索质量，减少噪声对最终生成结果的影响呢？

检索噪声指召回的文本块中包含了与问题不相关或弱相关的信息，混入提示词后误导模型，产生错误或偏离的答案。

优化检索质量的方法：

  • 优化分块策略：避免过长块引入噪声。

  • 混合检索：结合向量检索和关键词检索，举个栗子，BM25+向量加权，提升召回相关度。

  • 重排（Rerank）：对初步召回的结果使用交叉编码器模型进行二次排序，将最相关的排在前面。

  • 设置相似度阈值：过滤掉相似度得分低于阈值的块。

  • 查询改写：对用户问题进行改写后再检索，提高召回质量。

  • 提示工程：在prompt中明确告知模型只基于提供的上下文回答，忽略无关信息。

什么是向量数据库？它与传统的关系型数据库MySQL和搜索引擎Elasticsearch的核心区别是什么

向量数据库是一种专门用于存储和检索高维向量数据的数据库，它支持基于向量相似性的近似最近邻搜索。 区别：

  • 数据模型：MySQL存储结构化表格数据，ES存储文档（JSON）并建立倒排索引，向量数据库存储向量及其元数据，核心操作是向量相似度计算。

  • 查询方式：MySQL基于精确匹配和范围查询，ES基于倒排索引的文本匹配，向量数据库基于距离度量（余弦相似度or欧氏距离）进行近似最近邻搜索。

  • 索引结构：MySQL使用B+树，ES使用倒排索引，向量数据库使用专门为高维空间设计的索引算法以牺牲微小精度换取高速检索。

什么是Function Calling（工具调用）？

  1. Function Calling是一种让大语言模型能够根据用户意图，智能地选择并调用外部函数的能力。模型输出一个结构化的调用请求，包含函数名和参数，后端系统执行该函数并返回结果。

  2. 在AI Agent中，Function Calling使Agent具备行动能力。让Agent可以实时获取最新信息并执行业务逻辑

在模型训练或微调的角度，Function Call的能力是如何被赋予给大模型的？

Function Call能力主要通过后期训练阶段的指令微调（SFT）和偏好优化（DPO）注入。具体步骤：

  1. 构造训练数据：收集包含用户请求-函数描述-模型应该输出的函数调用的对话样本。主包举个栗子哈，比如说，用户问北京天气怎么样，我们要同时提供get_weather函数的描述，标注模型应该输出get_weather(location="北京")。

  2. SFT微调：让模型在大量这样的样本上学习，使其学会根据用户意图和可用函数生成正确的调用格式。

  3. 增强泛化：通过多样化的函数和请求让模型学会何时不调用（直接回答）、何时调用以及如何组合多个调用。

  4. 偏好优化：使用人类反馈或AI反馈DPO进一步优化，让模型更倾向于正确的调用行为。

当用户请求需要调用多个工具或进行复杂推理才能完成时（举个栗子，比如说"帮我订明天去北京的机票和酒店"），Agent的工作流程应该是怎样的

这类复杂任务通常采用ReAct模式（Reason+Act）推理加行动，即循环执行思考、行动、观察直到任务完成。

  • 模型分析用户请求，意识到需要分别订机票和酒店，且可能需要查询航班、酒店信息，以及进行预订操作。

  • 模型决定调用第一个工具，search_flights(destination="北京", date="明天")，生成调用请求。后端执行并返回航班列表。

  • 模型看到航班列表，继续思考需要选择一个航班，然后调用预订工具。

  • 模型根据观察继续思考，依次调用search_hotels、book_flight、book_hotel等工具。

package com.psychic.agent.service;

import org.springframework.ai.vectorstore.VectorStore;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.stream.Collectors;

@Service
public class KnowledgeBaseService {

    private final VectorStore vectorStore;

    public KnowledgeBaseService(VectorStore vectorStore) {
        this.vectorStore = vectorStore;
    }

    /**
     * 检索与用户问题最相关的知识片段
     * @param query 用户问题
     * @param topK 返回数量
     * @return 相关文本列表
     */
    public List<String> retrieve(String query, int topK) {
        return vectorStore.similaritySearch(query, topK)
                .stream()
                .map(document -> document.getContent())
                .collect(Collectors.toList());
    }

    /**
     * 添加文档到知识库（管理员调用）
     * @param content 文档内容
     */
    public void addDocument(String content) {
        // 这里可以拆分长文本为多个片段，并生成向量后存储
        vectorStore.add(List.of(content));
    }
}

package com.psychic.agent.service;

import com.psychic.agent.entity.PsychologicalReport;
import com.psychic.agent.repository.PsychologicalReportRepository;
import org.springframework.ai.chat.ChatClient;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.stereotype.Service;

@Service
public class PsychologicalService {

    @Autowired
    @Qualifier("ollamaChatClient")
    private ChatClient ollamaClient;

    @Autowired
    private PsychologicalReportRepository reportRepository;

    @Autowired
    private McpExcelService mcpExcelService;

    @Autowired
    private McpEmailService mcpEmailService;

    /**
     * 分析用户消息，返回风险等级（high/medium/low）和情绪标签
     * @param userId 用户ID
     * @param message 用户消息
     * @return 风险等级
     */
    public String analyzeAndMonitor(Long userId, String message) {
        // 调用微调后的模型进行心理状态识别（微调模型名称已在 ollama 中导入，如 "qwen-psych:7b"）
        String prompt = "请分析以下消息的心理状态，返回 JSON 格式：{\"risk\": \"high|medium|low\", \"emotion\": \"xxx\", \"confidence\": 0.xx}\n用户消息：" + message;
        String response = ollamaClient.call(prompt);

        // 解析响应（实际应使用 JSON 库，此处简化）
        String risk = extractRisk(response);
        String emotion = extractEmotion(response);
        double confidence = extractConfidence(response);

        // 保存心理报告
        PsychologicalReport report = new PsychologicalReport();
        report.setUserId(userId);
        report.setContent(message);
        report.setRiskLevel(risk);
        report.setEmotion(emotion);
        report.setConfidence(confidence);
        reportRepository.save(report);

        // 通过 MCP 服务写入 Excel
        mcpExcelService.writeReport(report);

        // 高风险时发送邮件
        if ("high".equals(risk)) {
            mcpEmailService.sendAlert(userId, "学生心理状态高风险，请及时关注");
        }

        return risk;
    }

    // 简单解析示例，实际使用 Jackson
    private String extractRisk(String response) {
        if (response.contains("\"risk\":\"high\"")) return "high";
        if (response.contains("\"risk\":\"medium\"")) return "medium";
        return "low";
    }

    private String extractEmotion(String response) {
        // 正则或 JSON 解析
        return "neutral";
    }

    private double extractConfidence(String response) {
        return 0.85;
    }
}

聊天服务 ChatService.java

package com.psychic.agent.service;

import com.psychic.agent.entity.ChatMessage;
import com.psychic.agent.repository.ChatMessageRepository;
import org.springframework.ai.chat.ChatClient;
import org.springframework.ai.chat.messages.Message;
import org.springframework.ai.chat.messages.SystemMessage;
import org.springframework.ai.chat.messages.UserMessage;
import org.springframework.ai.chat.prompt.Prompt;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.beans.factory.annotation.Qualifier;
import org.springframework.stereotype.Service;
import reactor.core.publisher.Flux;

import java.util.List;
import java.util.stream.Collectors;

@Service
public class ChatService {

    @Autowired
    @Qualifier("ollamaChatClient")
    private ChatClient ollamaClient;

    @Autowired
    private KnowledgeBaseService knowledgeBaseService;

    @Autowired
    private ChatMessageRepository chatMessageRepository;

    @Autowired
    private PsychologicalService psychologicalService;

    /**
     * 流式对话，整合 RAG + 心理监控
     * @param userId 用户ID
     * @param query 用户输入
     * @return 流式回答
     */
    public Flux<String> streamChat(Long userId, String query) {
        // 1. 心理监控（异步非阻塞，不影响流式返回）
        psychologicalService.analyzeAndMonitor(userId, query);

        // 2. RAG 检索相关知识
        List<String> contexts = knowledgeBaseService.retrieve(query, 3);
        String context = String.join("\n", contexts);

        // 3. 构建系统消息（包含检索到的知识）
        String systemPrompt = "你是一位专业的心理关怀助手。请基于以下知识库内容回答问题，如果知识库中没有相关信息，请如实告知。\n知识库：\n" + context;
        SystemMessage systemMessage = new SystemMessage(systemPrompt);

        // 4. 获取历史消息（最近 5 条）
        List<ChatMessage> history = chatMessageRepository.findTop5ByUserIdOrderByCreateTimeDesc(userId);
        List<Message> messages = history.stream()
                .map(msg -> new UserMessage(msg.getContent()))
                .collect(Collectors.toList());
        messages.add(0, systemMessage);
        messages.add(new UserMessage(query));

        // 5. 调用大模型，流式返回
        Flux<String> response = ollamaClient.stream(new Prompt(messages))
                .map(chunk -> chunk.getResult().getOutput().getContent());

        // 6. 异步保存用户消息和助手回复（这里简化，实际应收集完整回复后保存）
        ChatMessage userMsg = new ChatMessage();
        userMsg.setUserId(userId);
        userMsg.setContent(query);
        userMsg.setRole("user");
        chatMessageRepository.save(userMsg);

        return response;
    }

@RestController
@RequestMapping("/api/chat")
public class ChatController {

    @Autowired
    private ChatService chatService;

    @PostMapping("/stream")
    public Flux<String> streamChat(@AuthenticationPrincipal UserDetails userDetails,
                                   @RequestParam String query) {
        Long userId = extractUserId(userDetails); // 实际从 UserDetails 获取
        return chatService.streamChat(userId, query);
    }

        return ((CustomUserDetails) userDetails).getUserId();
    }
}
